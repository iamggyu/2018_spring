[LAB 2] RTL and Sequential Logic [Edited: 2018.03.21]

Pyeongsu Park (pyeongsu@snu.ac.kr)

Eunjin Baek (ebaek@snu.ac.kr)



[Notice]

2018.03.21: Reads should be asynchronous for the later projects, so we change the guideline (marked as underline) for the RF implementation. We do not grade asyncrhonous read in the testbench. However, you need to change the code in the later project if you design the RF as synchronous read.

Overview

The purpose of this project is to understand (1) what is register transfer level (RTL) and (2) how to design a sequential logic in Verilog.
The assignment covers them by implementing a "010 detector" and a "register file".



Goal

- Understand what is register transfer level (RTL) in Verilog
- Learn how to design sequential logic in Verilog



Assignment Overview

The assignment for this week is to design and implement (1) a 010 detector and (2) a synchronized 2-read 1-write register file.

Through this assignment, you can learn how to write down your sequential logic in Verilog, and it is a fundamental step for your successful CPU prototyping.

If you don't know how to start, refer to the material for this week; It contains how to design an FSM and a synchronous circuit.
 

 

Assignment 2-1: 010 detector

The input of the sequence detector is a binary stream, which consists of only 1 and 0.
The sequence detector's output is 1 when the recent three bits are "010".

Below is the example of 101 detector's input and output.

Input:   0000001010111001101
Output: 0000000101000000000

One bit per cycle is given as input, and output, which is 1 bit, should be changed before the next clock positive edge.

Of course, 1-bit clock is also an input of the detector.

* You should draw your FSM diagram on your report *

 

 

Assignment 2-2: Register File

A register file is a fundamental building block of a CPU.
Simply speaking, a register file is a collection of registers, which saves temporal values and program contexts.
There are many design choices for register files, but we decided to give a simple one as an assignment: a 2-port-read and 1-port-write register file.

 

Below is the block diagram of our register file.


Its inputs consist of (1) 1-bit write signal, (2) 1-bit clock, (3) three 2-bit addresses (two for reads and one for write), and (4) a 16-bit data that will be written to the register specified by address3. (5) and reset_n;

As outputs, the RF generates two 16-bit data specified by two read addresses.

The Data1 is for Addr1, Data2 is for Addr2, and Data3 is for Addr3.
In other words, Data1 is from the register specified by Addr1, Data2 is from the register specified by Addr2, and Data3 is written to the register specified by Addr3.
 

When the reset_n is low, all the register values set to 0.

If the write signal is enabled, or 1, the Data3 should be written at the next positive clock edge.
In other words, the write should be synchronized to the clock.


For example, if the "write" is initially turned on and becomes off before the positive clock edge, then the data should not be written.
If the "write" is turned off, of course, then there should be no write.
However, reads should be asynchronous to the clock; data1 and data2 should be changed immediately when the addr1 and addr2 are changed.

 

You need to use the smallest number of registers for 2-bit addresses in this assignment. If not, you lose some points.

You don't have to set the initial values of registers.

There should be no errors in your RF because you need to use this register file for the single cycle CPU.



 

 

Grading

Design: 010 detector

You need to design your 010-detector with an FSM. Your code should be written based on the FSM you drew in the report.

If there is a mismatch, then you cannot get a full credit.


Functionality: correct results on our testbench

In this week, we give a sample testbench that has simple patterns for the detector and RF. However, we may change the patterns for the grading.

We recommend testing your code for various patterns (make your own!).

It is better if your testbench covers lots of possible cases. 

We will grade your source codes with our own testbench that covers numerous situations.
You should remember that RF's write is synchronized to the clock, as mentioned above.
 

Code review

Your code should be easy to read. If not, you may get some penalties.

- You should provide the explanations of module's inputs, outputs, and functionality.
- All the variables, parameters, and file names should be self-explanatory. It should be neither "reg a", "wire wire1" nor "helloWorld.v".
- You stick to use a consistent code style. 

- Do not mix blocking and non-blocking assignment in a single always block.

- Do not use delay (e.g., #10) to correct the timing. Delays are only for testing.


Report

Please refer to "Lab1_How_to_wirte_your_report.pdf".
I'll follow the previous guideline for this week. 
In addition to this, you should explain how you designed 010-detector with a clear FSM.
